> 5축 웹 리서치 워크플로(10 에이전트, ~625k tokens) + 4개 핵심 주장 교차검증.

## Executive Summary

GOAL RESTATED: a general-purpose, true-W4A4 NVFP4 (E2M1 4-bit weights AND 4-bit activations, 16-element micro-blocks with E4M3 block scale + per-tensor FP32 global scale) PTQ + calibration toolkit for five ViT/vision-encoder families on 8x RTX PRO 6000 Blackwell (sm_120, CC 12.0, CUDA 13.1, driver 590.48.01, 96GB each). Environment confirmed live via nvidia-smi/nvcc; uv present; project dir empty (greenfield).

HEADLINE DECISIONS:
1. DO NOT write your own mma.sync block-scaled FP4 GEMM from scratch first. The dedicated NVFP4 atom (mma.sync.aligned.kind::mxf4nvf4.block_scale.scale_vec::4X.m16n8k64...ue4m3) is hardware-real on sm_120, but the per-lane scale-factor (SF) fragment layout for m16n8k64 FP4-E2M1 is UNDOCUMENTED in the PTX ISA (NVIDIA forum thread #364020 open as of Mar 2026), making a hand-rolled kernel high-risk. Reuse a collective-builder kernel instead.
2. RECOMMENDED GEMM PATH: build a thin Python-callable dense W4A4 NVFP4 GEMM op by reusing CUTLASS example 79 (79_blackwell_geforce_gemm: 79a nvfp4->bf16, 79b nvfp4->nvfp4) compiled with the alignment fix from CUTLASS issue #2906 and the sm_120a target, OR consume FlashInfer's b12x backend (flashinfer.gemm.mm_fp4, backend='b12x' or 'auto') which is the only path that "auto"-selects correctly on sm_120. Treat torch._scaled_mm FP4 as NOT-validated on sm_120 (torchao docs only gate on "SM100+" and benchmark only B200; do not assume it routes to a correct sm_120 kernel).
3. PTQ FRONTEND: reuse NVIDIA TensorRT Model-Optimizer (modelopt mtq.quantize + NVFP4_DEFAULT_CFG) as the calibration/observer engine OR torchao's NVFP4Tensor subclass primitives. Both already implement the correct two-level scaling math (global_scale = amax/(6.0*448.0); per-block = block_amax/6 clamped [2^-9,448] cast E4M3). You will write the ViT-specific wrapper yourself — no turnkey ViT NVFP4 toolkit exists (PTQ4ViT/FQ-ViT are INT8 and pre-Blackwell).
4. BIGGEST RISK: there is a HARDWARE-CAPABLE vs USABLE-FROM-PYTHON gap. On this exact stack (CUDA 13.1) the canonical FlashInfer cutlass/cudnn backends were reported BROKEN on RTX PRO 6000 (issue #2577 still open: cutlass returns all-zeros, cudnn raises cudnnGraphNotSupportedError) and CUTLASS 79a crashed with misaligned-address on CUDA 13.1 (issue #2906, community alignment fix not upstream-merged). You MUST stand up a numeric-correctness gate (FP4 GEMM vs BF16 reference) on YOUR GPU before building anything on top.
5. ACCURACY RISK: no published NVFP4 PTQ study evaluates ViTs; naive W4A4 on vision encoders collapses to ~0.1% (RegCache Table 2: SigLIP2-B 78.47->0.14, DINOv2-B 83.26->0.00). Plan for mixed-precision (keep attention QK/AV, first+last 2 blocks, patch-embed conv, norms, heads, and pooling in BF16) and budget for QAT/QAD as the likely-required step for vision-language encoders.


## Kernel Landscape (sm_120)

WHAT EXISTS AND IS USABLE ON sm_120 TODAY (in descending reliability):

(A) MARLIN W4A16 (vLLM): the "it works and is fast" fallback (~46-85 tok/s on real models). BUT THIS IS NOT W4A4 — it dequantizes FP4 weights to BF16/FP16 and does the matmul in 16-bit. It does NOT use the FP4 tensor cores for compute. Reporting Marlin as "NVFP4 working" misrepresents the W4A4 goal. Use only as a correctness/perf baseline, never as the deliverable.

(B) FlashInfer b12x backend (NVFP4-only, CuTe-DSL): flashinfer.gemm.mm_fp4(a, b, a_descale, b_descale, out_dtype=bf16, block_size=16, backend='auto'|'b12x'). On sm_120 'auto' prefers b12x->cutlass->cudnn. This is the most promising REUSE path for a true W4A4 dense GEMM. CAVEATS: flashinfer-cubin ships ZERO precompiled sm_120/121 cubins (issue #3294) so b12x relies on JIT/CuTe-DSL compilation at runtime — expect first-call compile latency and a working nvcc/CUTLASS toolchain. The older cutlass/cudnn backends were broken on sm_120 (issue #2577, still open). Pin a recent FlashInfer (>=0.6.7 / 0.6.11-era).

(C) CUTLASS example 79 (79_blackwell_geforce_gemm) C++ collective builders: arch=cutlass::arch::Sm120, operands cutlass::nv_float4_t<cutlass::float_e2m1_t>, SF via Sm1xxBlkScaledConfig, output bf16 (79a) or nvfp4 (79b). Landed CUTLASS 3.9.0. This is the cleanest path to BUILD your own pybind/torch-extension dense GEMM. KNOWN BUG on YOUR stack: example 79a crashed with misaligned-address on RTX 5090 + CUDA 13.1 + CUTLASS 4.3.4 (issue #2906) — fix = add alignas(64) to TMA-descriptor Params and alignas(128)/16-byte alignment to SF smem (cute::array_aligned in sm120_blockscaled_mma_tma.hpp); after fix ~3043 GFLOPS. The fix is COMMUNITY-supplied, NOT upstream-merged as of the snapshot — you must apply it yourself.

(D) cuBLASLt block-scaled FP4 (CUDA 12.9+): CUDA_R_4F_E2M1 data + CUDA_R_UE4M3 scales (VEC16_UE4M3 scale mode), 16-element NVFP4 blocks. Requires PTX 8.7+. NVIDIA docs do NOT cleanly confirm sm_120 vs sm_100 for FP4 — treat as plausibly-supported-but-underdocumented; verify on-device.

(E) vLLM native sm_120 NVFP4: vLLM 0.13.0+ ships nvfp4_scaled_mm_sm120_kernels.cu and PR #40082 (MERGED ~May 2026) integrates FlashInfer b12x MoE+FP4 GEMM for SM120/121 (FlashInferB12xNvFp4LinearKernel). Select via --linear-backend flashinfer-b12x. Good for cross-checking your kernel against a production dense path, but vLLM is inference-serving, not a PTQ toolkit.

HARDWARE FACTS (load-bearing for any kernel work): sm_120 has NO tcgen05 and NO Tensor Memory (TMEM) — it uses SM8x-style warp-level synchronous mma.sync from RMEM. ALL sm_100 (B200) FP4 GEMM kernels are incompatible. Two PTX kinds: kind::mxf4nvf4 (dedicated NVFP4, m16n8k64, scale_vec::4X, UE4M3, block 16) and kind::mxf8f6f4 (mixed FP8/FP6/FP4, m16n8k32, scale_vec::1X/2X). GeForce/RTX-Pro constraints: no TMA multicast, cluster forced 1x1x1.

CRITICAL BUILD-FLAG GOTCHA: NVFP4 block-scaled MMA needs the architecture-specific virtual target sm_120a (or compute_120f for fast grouped/MoE tactics on CUDA 13.0). PyTorch auto arch-detection STRIPS the 'a'/'f' suffix (sm_120a->sm_120), silently breaking NVFP4 (pytorch issue #172807). Always set TORCH_CUDA_ARCH_LIST=12.0a (and use compute_120f if you later need MoE/grouped). This is load-bearing, not cosmetic.

MoE/GROUPED FP4: fragile and out-of-scope for ViTs (vision towers are dense, no MoE). Only Qwen3-VL's TEXT decoder is MoE — irrelevant to the ViT encoder. Ignore grouped-GEMM risk entirely for this toolkit.


## PTQ Recipe (two-level scaling)

NVFP4 TWO-LEVEL SCALING MATH (verified identical across modelopt, torchao, compressed-tensors; implement exactly this — DO NOT store amax_block/6 directly):
- Element grid: E2M1 magnitudes {0,0.5,1,1.5,2,3,4,6}, max=6.0. Block=16 elements. Block scale dtype=FP8 E4M3 (UE4M3, max=448). Plus one per-tensor FP32 global scale.
- Global encode scale: s_enc = (6*448)/amax_tensor, equivalently global_scale = amax/(6.0*448.0) depending on encode/decode convention (modelopt: global_scale = global_amax/(6.0*448.0)).
- Per-block: s_dec,b = amax_block/6; STORED block scale = e4m3(s_dec,b * s_enc) round-to-nearest-even (the multiply by s_enc BEFORE the E4M3 cast is what keeps the FP8 scale within 448 — this coupling is mandatory). torchao/modelopt: per_block_scale = block_amax/6.0 clamped to [2^-9, 448] then cast float8_e4m3fn.
- Dequant (done by HW tensor core): x ≈ code_E2M1 * block_scale_E4M3 * global_scale_FP32. Forgetting the global scale silently truncates dynamic range.
- DO NOT confuse with MXFP4 (block 32, E8M0 power-of-two scale, no FP32 global).

CALIBRATION DESIGN (static vs dynamic is split PER SCALE LEVEL, not per tensor):
- WEIGHTS: fully STATIC/offline — per-group(16) E4M3 scales + per-tensor FP32 global, baked into checkpoint. No data needed.
- ACTIVATIONS (the only reason calibration data exists): per-TENSOR FP32 global activation scale is STATIC (calibrated from data via amax/min-max observer), while per-16-element LOCAL block scales are DYNAMIC (computed online at inference from each block's live amax — Blackwell HW does this micro-scaling inside the GEMM). compressed-tensors encodes this as input_activations dynamic=LOCAL, observer='static_minmax'.

RECOMMENDED CALIBRATION ENGINE: modelopt mtq.quantize(model, NVFP4_DEFAULT_CFG, forward_loop). modelopt inserts TensorQuantizer into nn.Linear/nn.Conv, patches forward to fake-quant, runs your forward_loop over ~128-512 representative images, collects amax via reduce_amax(), then folds scales. config encodes num_bits=(2,1) [E2M1], block_sizes={-1:16}, scale_bits=(4,3) [E4M3 block scale]. Available algorithms: "max" (default), MSE (NVFP4MSECalibrator), percentile, "smoothquant", "awq_lite/awq_full/awq_clip". Alternative engine: torchao NVFP4Tensor.to_nvfp4 + NVFP4DynamicActivationNVFP4WeightConfig (model-agnostic quantize_()).

CALIBRATION ALGORITHM CHOICE FOR ViT: start with "max"/min-max but EXPECT it to be insufficient for activations. Use percentile or MSE clipping for post-LayerNorm/post-GELU activations (heavy-tailed). For W4A4 specifically the per-tensor global activation scale is dominated by a few extreme-norm CLS/register/background tokens — use per-token amax exclusion or percentile (e.g. 99.9%) rather than raw max, or the activation global scale will be blown out by outlier tokens.

OUTLIER HANDLING — NVFP4-SPECIFIC NUANCE (do not blindly port LLM tricks):
- Rotation (QuaRot fixed Hadamard / SpinQuant learned) CAN HURT NVFP4 with plain round-to-nearest (per "Bridging the Gap" arXiv 2509.23202). If you use Hadamard, MATCH the Hadamard size to the group size: Had16 for NVFP4 (vs Had128 for MXFP4's group-32). Apply rotation BEFORE micro-block grouping; fuse weight rotations offline, do activation rotations online.
- NVIDIA's QAD report (arXiv 2601.20088) warns the small 16-element block "neutralizes traditional outlier mitigation techniques" — SmoothQuant/rotation lose leverage when scaling is already fine-grained, and PTQ trails QAD by 20-30%.
- "Four Over Six" (arXiv 2512.02010): the DOMINANT FP4 error is NOT outliers but non-uniform mid-range step sizes (values at 66-100% of block max are poorly represented). Adaptive M=4/M=6 per-block scale selection fixes this. Consider implementing this for the softmax power-law problem (values near block max need fine resolution that FP4's coarse high-range steps lack).

EXPORT: modelopt export_hf_checkpoint produces HF-style packed FP4 + weight_scale (E4M3) + weight_global_scale + input_global_scale. compressed-tensors uses {weight_packed, weight_scale, weight_global_scale, input_global_scale}; modelopt/vLLM expect {weight, weight_scale_inverse, input_scale}. Pick ONE ecosystem — cross-loading needs a converter. Packing: E2M1 two-per-byte (q[...,1::2]<<4)|q[...,0::2], dtype torch.float4_e2m1fn_x2.

GEMM LAYOUT GOTCHA: the HW GEMM does NOT accept row-major scales. Scales MUST be pre-swizzled to the 128x4 layout (torchao to_blocked() / FlashInfer SfLayout.layout_128x4). Un-swizzled scales give WRONG results, not an error. K (contraction) must be divisible by 16; scale rows padded to 128.


## ViT-specific Concerns

WHY ViT ACTIVATIONS BREAK AT W4A4 (the problem is overwhelmingly activations, not weights):
1. Post-LayerNorm activations have severe inter-channel variation (some channel ranges >40x the median) — a single per-tensor scale clips wide channels or wastes resolution. NVFP4's per-16-block scaling helps localize this (an outlier consumes only its own block's range) but per the NVIDIA QAD report the small block "neutralizes traditional outlier mitigation," so block scaling alone is NOT a full fix.
2. Post-Softmax attention follows a power-law (mass near 0, tiny fraction near 1). Uniform/coarse FP4 grids destroy the few large attention weights that route information. This maps directly onto FP4's coarse high-range steps ("Four Over Six"). MITIGATION: keep the QK and post-softmax A*V matmuls in BF16 (the most fragile matmuls); do NOT quantize softmax to FP4.
3. Post-GELU (tanh-approx for SigLIP/SigLIP2/Qwen3-VL) is strongly asymmetric — symmetric FP4 handles it poorly. The fc2 input is the activation feeding from GELU; calibrate its scale carefully (percentile/MSE).

CLS / REGISTER / HIGH-NORM TOKEN OUTLIERS (first-class problem):
- DINOv2 grows high-norm artifact tokens (~10x norm, ~2% of tokens) on background patches; DINOv3 has 1 CLS + 4 register tokens; SigLIP/SigLIP2/Qwen3-VL-vision have NO CLS/register (use attention-pool probe / merger); V-JEPA2 has no CLS (attentive pooler).
- RegCache (arXiv 2510.04547) shows vision-encoder outliers arise in MIDDLE layers (blocks ~5-8) on PER-IMAGE-VARYING background tokens — UNLIKE LLMs' fixed-vocab BOS/SEP sinks. This means you cannot pre-identify outlier positions; you need robust per-token/per-block activation scaling and possibly prefix-register insertion (RegCache reduced CLIP max token norm 41.38->11.45, +44.5% at W4A4).
- A handful of extreme-norm tokens blows up any per-tensor activation global scale. STRONGLY prefer per-token amax handling when calibrating the activation global scale (exclude or percentile-clip the top tokens).

REALITY CHECK ON W4A4 FEASIBILITY:
- W4A4 "success" in the literature (RepQuant/DopQ-ViT high-70s/low-80s) is for SUPERVISED ImageNet CLASSIFIERS with special non-uniform softmax/LN quantizers — NOT vision encoders. Naive W4A4 collapses: FQ-ViT ~0.1%; RegCache Table 2 naive W4A4 RTN: SigLIP-B 76.05->0.13, SigLIP2-B 78.47->0.14, DINOv2-B 83.26->0.00.
- Vision-LANGUAGE encoders (CLIP zero-shot) at W4A4 remain essentially broken (~1.8% even with RegCache). For SigLIP2/Qwen3-VL-vision contrastive/VLM use, plan for QAT/QAD, not pure PTQ.
- The closest ViT-native FP4 evidence is MXFP4 (block-32) TRAINING (TetraJet) which needs QAT + oscillation control to reach near-FP. NVFP4 PTQ on ViTs is genuinely under-evaluated — your toolkit will be charting new territory; build strong per-layer accuracy diagnostics (cosine-sim of layer outputs vs FP) from day one.

MIXED-PRECISION RECIPE (mirror NVIDIA's NVFP4 production pattern + ViT PTQ practice):
- KEEP IN BF16: all attention QK^T and A*V matmuls; first 2 and last 2 transformer blocks; patch-embed conv (Conv2d/Conv3d — FP4 ranges too narrow, modelopt NVFP4 ONNX fails on CNNs); all LayerNorms; the classification/projection head; the pooling head (SigLIP MAP probe / V-JEPA2 attentive pooler / Qwen3-VL merger).
- QUANTIZE TO NVFP4 W4A4: the QKV-projection, attn-output-projection, MLP fc1 (up), MLP fc2 (down) Linear layers in the MIDDLE blocks. These are the bulk of compute and the safe targets.
- Sensitivity studies (Mix-QViT/LRP-QViT) rank the projection layer most important, MSA/GELU highly sensitive, early blocks > deep blocks — so deeper-block Linears tolerate W4A4 best.


## Target Models (per-family arch)

All five are standard PRE-NORM ViTs (LayerNorm before attn+MLP, residual after). NONE use RMSNorm in the encoder — RMSNorm appears ONLY in Qwen3-VL's TEXT decoder. Per-family notes that change the quant config:

SigLIP / SigLIP2:
- LayerNorm eps=1e-6. SEPARATE q_proj/k_proj/v_proj + out_proj. MLP fc1/fc2. Activation gelu_pytorch_tanh. NO CLS, NO register tokens — pools via SiglipMultiheadAttentionPoolingHead (learned self.probe). Learned interpolatable pos-embeds. NO LayerScale.
- SO400M backbone (SigLIP-SO400M, SigLIP2-SO400M): hidden=1152, MLP intermediate=4304 (mlp_ratio~3.74, NOT 4x — do not hardcode 4*hidden), layers=27, heads=16, head_dim=72 (NOT 64), patch=14 (SigLIP2-SO400M-patch14-384) or 16. SigLIP2-base: hidden=768, MLP=3072, 12 layers, 12 heads, patch=16.
- Patch-embed = Conv2d. Keep MAP pooling head + post_layernorm in BF16.

Qwen3-VL vision encoder (SigLIP2-style ViT):
- LayerNorm eps=1e-6 (norm1/norm2, NOT RMSNorm). FUSED qkv = nn.Linear(dim, 3*dim, bias=True) + proj output. MLP is PLAIN linear_fc1->gelu_tanh->linear_fc2 (NOT gated). Patch-embed = Conv3d (kernel [temporal=2,16,16]). Hybrid pos: learned nn.Embedding (bilinear-interp) + 2D rotary. NO CLS/register. NO LayerScale.
- 30B-A3B vision: depth=27, hidden=1152, intermediate=4304, heads=16, patch=16, out_hidden_size=2048, spatial_merge_size=2 (4 patches merged, token count /4 — merger input dim = hidden*merge^2). DeepStack injects ViT layers [8,16,24]. Dynamic seq length (NaFlex/grid_thw). The FUSED qkv means your per-layer weight mapping must branch vs the separate-qkv families.

DINOv2:
- LayerNorm eps=1e-6. SEPARATE query/key/value + 'dense' output. MLP fc1/fc2 (gelu, exact). LayerScale (lambda1) AFTER BOTH attn and MLP — per-channel learned multiply, creates per-channel dynamic range; can fold into preceding GEMM output scale but must not ignore. 1 CLS token; dinov2-with-registers adds 4 registers. Learned interpolatable pos-embeds.
- dinov2-large: hidden=1024, 24 layers, 16 heads, mlp_ratio=4, patch=14, image=518, use_swiglu_ffn=FALSE (plain fc1/fc2 — do NOT assume SwiGLU). dinov2-base: hidden=768, 12 layers, head_dim=64. High-norm artifact tokens are the key outlier hazard.

DINOv3 ViT:
- LayerNorm eps=1e-5 (DIFFERENT from the others — affects scale-folding into norm). SEPARATE q_proj/k_proj/v_proj with ASYMMETRIC bias (query_bias=True, key_bias=FALSE, value_bias=True — a naive all-or-nothing qkv-bias assumption breaks DINOv3) + o_proj. RoPE (rope_theta=100.0) on patch tokens only, no learned abs pos-embed. 1 CLS + 4 register tokens. LayerScale after attn+MLP. MLP configurable: plain up_proj/down_proj OR gated SwiGLU gate_proj/up_proj/down_proj (large variants e.g. ViT-7B/16 use SwiGLU — gated MLP means 3 GEMMs).
- ViT-S: hidden=384, 12 layers, 6 heads, patch=16. ViT-B: hidden=768.

V-JEPA2 / 2.1:
- LayerNorm eps=1e-6. SEPARATE query/key/value + 'proj' output. MLP fc1/fc2 (plain, gelu). Patch-embed = Conv3d (tubelet=2,16,16). 3D-RoPE over (time,H,W). NO CLS — pools via VJEPA2AttentivePooler (learned query tokens). NO LayerScale.
- ViT-g (vjepa2-vitg-fpc64-256): hidden=1408, 40 layers, 22 heads, mlp_ratio~4.36 (intermediate~6144), patch=16, 64 frames. Predictor is a separate small ViT (hidden=384, 12 layers) — quantize encoder only. V-JEPA 2.1 scales to ViT-G (~2B); treat 2.1 dims as paper-stated until you read the actual checkpoint config.json.

CROSS-CUTTING: attention logit scale is vanilla head_dim^-0.5 everywhere (no learned temperature in vision towers). Your nn.Linear-swap config MUST branch per family on: qkv fusion (fused for Qwen3-VL; separate elsewhere with names q_proj/query), output-proj name (out_proj/proj/dense/o_proj), MLP gating (gated only DINOv3-large), LayerScale presence (DINOv2/DINOv3 only), LN eps (1e-5 DINOv3 vs 1e-6 rest), and the non-4x MLP ratio for SO400M.


## Recommended Stack

CONCRETE BUILD APPROACH (greenfield, uv-managed, 8x sm_120 / CUDA 13.1):

ENV (use uv, per project policy):
- TORCH_CUDA_ARCH_LIST=12.0a is MANDATORY for any FP4 kernel build (pytorch #172807 strips the 'a' otherwise). If you later touch grouped/MoE, use compute_120f.
- PyTorch nightly/recent with CUDA 13.x wheels; CUTLASS 4.5+ checkout; FlashInfer >=0.6.7 (expect JIT compile of sm_120 cubins on first call — no precompiled cubins shipped, issue #3294); modelopt (>=0.43) for calibration; transformers (HF for all 5 model families); timm for any timm ViTs.

PHASE 0 — GEMM CORRECTNESS GATE (do this FIRST, before any toolkit code):
On the actual RTX PRO 6000, run a standalone W4A4 NVFP4 dense GEMM and compare against a BF16 reference (rel-error < a threshold). Try in order: (a) FlashInfer flashinfer.gemm.mm_fp4(backend='b12x'); (b) CUTLASS example 79b built with sm_120a + the issue-#2906 alignment fix (alignas(64) on TMA Params, alignas(128)/array_aligned on SF smem); (c) vLLM cutlass_scaled_fp4_mm via --linear-backend. EXPECTED FAILURES on this CUDA 13.1 stack: FlashInfer cutlass backend may return all-zeros (#2577), cudnn raises cudnnGraphNotSupportedError, CUTLASS 79a may crash misaligned-address pre-fix. DECISION GATE: whichever path produces correct numerics on-device becomes your GEMM. If NONE pass, fall back to building the CUTLASS-79 collective-builder kernel + a torch custom-op binding yourself (the cuda / cute-dsl-ref skills are available for this). Do NOT hand-roll raw mma.sync — the SF fragment layout is undocumented (#364020).

PHASE 1 — PTQ FRONTEND (reuse, do not rebuild the math):
Choose ONE engine and commit to its checkpoint format:
- Option A (recommended for ViT generality): torchao NVFP4Tensor + NVFP4DynamicActivationNVFP4WeightConfig. Model-agnostic quantize_() over any nn.Linear tree; you control which Linears get swapped. Cleanest reference for the two-level math; but swap torchao's internal torch._scaled_mm call for YOUR Phase-0-validated GEMM (override nvfp4_linear) since torch._scaled_mm FP4 is unvalidated on sm_120.
- Option B: modelopt mtq.quantize + NVFP4_DEFAULT_CFG for calibration/observers, then export and run the GEMM yourself. Best calibration algorithm menu (MSE/percentile/AWQ).
Build a per-family quant-config map (the targetModels branching) that selects which Linears to quantize and which to keep BF16 (attention QK/AV, first/last 2 blocks, patch-conv, norms, heads, pooling).

PHASE 2 — CALIBRATION HARNESS:
forward_loop over ~128-512 representative images per model family (ImageNet-style for SigLIP/DINOv2/DINOv3; video clips for V-JEPA2; image+text for SigLIP2/Qwen3-VL). Collect per-tensor activation global scales (percentile/MSE, NOT raw max, to survive CLS/register/background-token outliers). Weights fully static.

PHASE 3 — ACCURACY DIAGNOSTICS (build early, it is your steering wheel):
Per-layer cosine-similarity of FP4 vs BF16 outputs; end-task metric (ImageNet top-1 / zero-shot acc / linear-probe). Use these to drive mixed-precision: promote any Linear whose quantization tanks downstream metric back to BF16. Expect to iterate on outlier mitigation (Had16 rotation — test carefully, it can hurt NVFP4; RegCache-style prefix registers for vision encoders) and to conclude that contrastive/VLM encoders need QAT/QAD.

MULTI-GPU: 8 GPUs are for PARALLEL calibration/eval throughput (run different model families / different calibration configs concurrently), NOT for a single quantized model — these ViT encoders fit easily in one 96GB card. No tensor-parallel needed.

REUSE-vs-BUILD SUMMARY: REUSE the PTQ math (torchao/modelopt) and the calibration loop (modelopt). REUSE the dense GEMM if Phase-0 passes (FlashInfer b12x or CUTLASS-79+fix). BUILD: the ViT-specific per-family wrapper/config, the calibration harness, the accuracy diagnostics, and ONLY IF Phase-0 fully fails, a CUTLASS-79-based torch custom-op GEMM. There is genuinely no existing ViT NVFP4 toolkit to fork.


## Open Questions

- GEMM-VALIDATION (BLOCKER): Does ANY Python-callable W4A4 NVFP4 dense GEMM produce numerically-correct output on THIS exact RTX PRO 6000 + CUDA 13.1 + driver 590 stack? The research has FlashInfer #2577 (all-zeros/cudnn-error, still open) and CUTLASS #2906 (misaligned crash, community fix not upstream) both on CUDA 13.1. This must be answered empirically in Phase 0 before any toolkit work; it determines reuse-vs-build for the kernel.
- Is the CUTLASS-79 alignment fix (alignas-64 TMA Params, array_aligned SF smem) sufficient on CUDA 13.1 / CUTLASS 4.5+, or are there additional sm_120a-specific bugs? Need to compile and run 79b on-device.
- Does FlashInfer b12x JIT-compile successfully on this box (no precompiled sm_120 cubins ship, #3294) given CUDA 13.1 nvcc, and what is the first-call compile latency / caching behavior?
- Can torchao's nvfp4_linear be cleanly overridden to call a custom validated GEMM instead of torch._scaled_mm, without fighting the tensor-subclass dispatch? Or is it easier to build the wrapper directly on modelopt's quantizer outputs?
- ACCURACY (research gap): No published NVFP4 PTQ result exists for ANY ViT. What W4A4 accuracy is actually achievable per-family with mixed-precision PTQ alone (before QAT)? Especially: can supervised DINOv2/v3 reach usable linear-probe accuracy, and is SigLIP2/Qwen3-VL contrastive/zero-shot recoverable at all without QAT (RegCache says CLIP W4A4 ~1.8%)?

### [CONFIRMED] Naive W4A4 (4-bit weights AND 4-bit activations) with only per-block scaling causes large accuracy loss on Vision Transformers unless outlier mitigation (rotation/Hadamard such as QuaRot/SpinQuant, or SmoothQuant-style scaling) and/or keeping a few sensitive layers in higher precision is applied.

Multiple independent primary sources confirm every component of the claim, with the strongest evidence being ViT-specific.

(1) ViT-SPECIFIC, NAIVE W4A4 COLLAPSES. The peer-reviewed-style paper "Activation Quantization of Vision Encoders Needs Prefixing Registers" (RegCache, arXiv:2510.04547v4, POSTECH/Dankook/Google, Mar 2026) gives Table 2 zero-shot ImageNet-1k accuracy for naive W4A4 (round-to-nearest) on vision encoders that essentially collapses to chance: CLIP-B/16 FP32=68.32 -> Naive W4A4 = 0.09; OpenCLIP-B/16 70.22 -> 0.10; SigLIP-B/16 76.05 -> 0.13; SigLIP2-B/16 78.47 -> 0.14; DINOv2-B/14 83.26 -> 0.00. The paper explicitly states "PTQ methods for ViT-based vision encoders still suffer from significant performance degradation under low-bit quantization" and attributes it to "outlier activations -- i.e., a small number of extremely large activations" concentrated in a few channels of later layers (notably the FC2 MLP layers in middle blocks, Fig. 3). It shows outlier-aware methods recover large accuracy at W4A4 (e.g., FIMA-Q W4A4 reaches 50.41 on CLIP, 78.48 on SigLIP2; RegCache, which removes/relocates outlier "register" tokens, further improves nearly every baseline). This directly supports BOTH the "outlier mitigation" and the "the few sensitive layers" parts of the claim.

(2) ROTATION/HADAMARD (QuaRot, SpinQuant) on transformers. SpinQuant (arXiv:2405.16406) states "rotating activation or weight matrices helps remove outliers and benefits quantization." For 4-bit weights+activations+KV it narrows the gap to full precision to ~2.9 points on LLaMA-2 7B, beating SmoothQuant by 25.0 points and LLM-QAT by 19.1 points, and improving over QuaRot's fixed Hadamard rotations. QuaRot (Hadamard rotation) is reported to retain >99% of full-precision accuracy at 4-bit. These are LLMs, but establish the rotation/Hadamard outlier-mitigation mechanism the claim cites. Rotation-based W4A4 on other transformer families is similarly necessary: ConvRot (arXiv:2512.03673) shows naive W4A4 on FLUX.1-dev diffusion transformer degrades FID 10.07->12.32 and that Hadamard rotation suppresses activation outlier magnitude (e.g., 14.48->9.26).

(3) SMOOTHQUANT-STYLE SCALING and KEEPING SENSITIVE LAYERS/CHANNELS HIGHER PRECISION. RepQ-ViT (arXiv:2212.08254) confirms ViT PTQ schemes "suffer from non-trivial accuracy degradation, especially in low-bit cases" and that the core obstacles are "post-LayerNorm activations with severe inter-channel variation" (channel outliers) and "post-Softmax activations with power-law features"; it applies channel-wise and log-sqrt(2) (outlier-aware) quantization to bring W4A4 ViT PTQ "to a usable level." OSC (arXiv:2604.12782) demonstrates naive W4A4/MXFP4 drops ~6.09 points on Qwen3-8B and recovers to ~2.19 by keeping persistently-outlier channels in 16-bit precision (i.e., mixed precision for sensitive components) -- the same higher-precision mitigation strategy.

CAVEAT / why not 100% unqualified: The claim says "only per-block scaling." The strongest ViT numbers (RegCache Table 2) use a naive per-token/RTN baseline rather than an explicitly labeled "per-block" (MXFP4/NVFP4-style) scheme, and the explicit per-block W4A4 collapse evidence (OSC MXFP4) is on LLMs, not ViTs. NVFP4/MXFP4 per-block formats can partially help vs per-tensor, but the consistent finding is that finer block scaling alone does not solve the dominant channel/token outlier problem -- mitigation (rotation, scaling, or higher-precision sensitive layers) is still required. The directional claim is robustly supported; the specific phrase "per-block scaling" is slightly stronger than the cleanest ViT primary source, hence the claim is confirmed in substance with this minor scoping nuance noted.

## Research by Dimension (raw, with sources)

### 2025-2026 native NVFP4/FP4 (E2M1) W4A4 matmul kernel support on NVIDIA consumer/workstation Blackwell sm_120 / CC 12.0 (RTX 5090, RTX PRO 6000 Blackwell): hardware capability vs Python usability across PTX MMA, CUTLASS, cuBLASLt, PyTorch/torchao/FlashInfer, and vLLM/TensorRT-LLM.

The 5th-gen tensor core on consumer/workstation Blackwell (sm_120 / sm_120a, compute capability 12.0; RTX 5090, RTX PRO 6000 Blackwell) DOES expose native FP4 E2M1 block-scaled MMA, but via the classic SM8x-style WARP-LEVEL synchronous `mma.sync` path sourcing operands from registers (RMEM), NOT the tcgen05/TMEM asynchronous tensor-memory path that is exclusive to datacenter sm_100 (B200/B300). Two PTX instruction families exist: `mma.sync.aligned.kind::mxf8f6f4.block_scale...m16n8k32...` (mixed FP8/FP6/FP4 inputs, shape m16n8k32, scale_vec::1X/2X) and the dedicated NVFP4 path `mma.sync.aligned.kind::mxf4nvf4.block_scale.scale_vec::4X.m16n8k64.row.col.f32.e2m1.e2m1.f32.ue4m3` (shape fixed at m16n8k64, UE4M3 scale, block size 16). Scale factors are warp-distributed via optional selector operands `{byte-id-a, thread-id-a}` / `{byte-id-b, thread-id-b}`; the precise per-lane fragment layout for the SF operands is poorly documented (a March 2026 NVIDIA forum thread notes the FP4-E2M1 SF fragment diagram is missing from the PTX ISA, making hand-written kernels error-prone).

CUTLASS is the most mature usable path: SM120 GeForce Blackwell blockscaled DENSE GEMM, the NVFP4 collective builders (`cutlass::arch::Sm120`, operands `cutlass::nv_float4_t<cutlass::float_e2m1_t>`, UE4M3 scale via `Sm1xxBlkScaledConfig`, BF16 or FP4 output), and example 79 `79_blackwell_geforce_gemm` (79a nvfp4->bf16, 79b nvfp4->nvfp4, 79d grouped/MoE) all landed in CUTLASS 3.9.0 (2025-04-24). SM120 grouped/blockscaled GEMM only stabilized much later: blockscaled MoE API in 4.3.0 (2025-11-21) and SM120 mixed-input blockscaled GROUPED GEMM in 4.5.0 (2026-05-01). Even the dense path had real bugs: example 79a crashed with a misaligned-address fault on RTX 5090 (CUDA 13.1, CUTLASS 4.3.4) due to missing alignas on TMA descriptors and SF smem; with the fix it passed at ~3043 GFLOPS.

cuBLASLt exposes block-scaled FP4 matmul (introduced in cuBLAS 12.9, CUDA Toolkit 12.9) using data type `CUDA_R_4F_E2M1` with `CUDA_R_UE4M3` (16-element blocks, NVFP4) or `CUDA_R_E8M0`/UE8M0 (32-element blocks, MXFP4) scales, configured via matmul-descriptor scale-mode attributes whose enum members are `VEC16_UE4M3`, `VEC32_UE8M0`, plus `OUTER_VEC_32F`, `VEC128_32F`, `BLK128x128_32F`. Requirement is a device supporting PTX 8.7+; NVIDIA's docs do not cleanly separate sm_100 from sm_120 for FP4, so sm_120 cuBLASLt FP4 should be treated as plausibly-supported-but-underdocumented rather than confirmed-working.

The Python/framework picture is the critical hardware-capable vs usable-today gap. Through early-to-mid 2026, native FP4 on sm_120 from Python was largely broken: FlashInfer 0.6.3 (CUDA 13) had ALL mm_fp4 backends fail on RTX PRO 6000 (cudnn: cudnnGraphNotSupportedError; cutlass: silently returns all zeros; trtllm: BackendSupportedError capability 120 unsupported). The state then improved: FlashInfer added a dedicated `b12x` backend (NVFP4-only, CuTe DSL) that `auto` now prefers on SM120 (b12x -> cutlass -> cudnn), plus a b12x CuTe DSL fused MoE wrapper (B12xMoEWrapper), surfacing in the 0.6.1x series. The hardest failure is MoE grouped GEMM: CUTLASS TMA-warp-specialized grouped block-scaled FP4 kernels fail to initialize on sm_120 with compute_120a and produce garbage tokens; a community fix used FlashInfer SM120 patches plus building with compute_120f on CUDA 13.0 to enable the fast grouped tactics, reaching ~39 tok/s native FP4 (vs Marlin W4A16 dequant fallback at ~46-50 tok/s). In vLLM the sm_120 NVFP4 dense-linear kernels exist (nvfp4_scaled_mm_sm120_kernels.cu) and cutlass_scaled_mm_supports_fp4(120) returns True, but backend selection historically mis-routed sm_120 to the Marlin weight-only fallback (the get_mxfp4_backend check only matched capability family 100); native MoE for SM_120 is being added through PRs #24968 (CUTLASS, long-running/draft with illegal-memory-access in real inference) and #40082 (integrating FlashInfer b12x MoE+FP4 GEMM for SM120/121). TensorRT-LLM standalone FP4 on RTX 5090/sm_120 has lagged (long-standing feature requests #3529/#5018; TRT-LLM trtllm backend explicitly rejects capability 120 in FlashInfer). PyTorch core torch._scaled_mm + torchao expose nvfp4/mxfp4 with dtypes torch.float4_e2m1fn_x2 (data), torch.float8_e4m3fn (nvfp4 scale), torch.float8_e8m0fnu (mxfp4 scale), but the official torchao Blackwell blog benchmarks only B200/sm_100 and states 'CUDA capability of at least 10.0' without validating sm_120; the fast NVFP4 path there routes through MSLK, not a guaranteed sm_120 kernel. Net: sm_120 is hardware-capable for native FP4 W4A4 via warp-level mma.sync; dense GEMM is usable from Python today through CUTLASS/FlashInfer-b12x on recent CUDA 13.0+/CUTLASS 4.5+ stacks, while MoE/grouped FP4 only became correct very recently and remains fragile, with Marlin W4A16 weight-only dequant as the reliable fallback.

### NVFP4 post-training quantization (PTQ) end-to-end: numeric format & two-level scaling math; NVIDIA TensorRT Model Optimizer (modelopt) API/calibration/export; llm-compressor & compressed-tensors NVFP4 scheme; W4A4 outlier handling (Hadamard/QuaRot/SpinQuant, SmoothQuant); static vs dynamic activation quantization.

NVFP4 is a two-level (double) block-scaled 4-bit floating-point format introduced for NVIDIA Blackwell. The element type is FP4 E2M1 (1 sign, 2 exponent, 1 mantissa), whose representable magnitudes are exactly {0, 0.5, 1, 1.5, 2, 3, 4, 6} (max = 6). Elements are grouped into micro-blocks of 16 values, each block sharing one FP8 E4M3 scale (max representable = 448), and the whole tensor additionally carries one FP32 per-tensor "global" scale. The two-level math (from the NVIDIA NVFP4 pretraining paper, Appendix B, and matched by compressed-tensors source) is: global encode scale s_enc = (6 * 448) / amax_tensor (compressed-tensors writes this as global_scale = FP8_E4M3.max * FP4_E2M1.max / amax). Per block, the decode scale is s_dec,b = amax_block / 6; this is multiplied by s_enc and rounded to E4M3 to get the stored block scale: s_dec,b,e4m3 = e4m3(s_dec,b * s_enc) (round-to-nearest-even). To quantize, modelopt/compressed-tensors recover a higher-precision local encode scale s_enc,b = 1 / (fp32(s_dec,b,e4m3) * s_dec) and quantize each element x_hat_i = q(x_i * s_enc,b) onto the E2M1 grid (RNE). Dequant at the Tensor Core is x ≈ code_E2M1 * block_scale_E4M3 * global_scale_FP32. The 6.0 clamp is the FP4 max used to normalize each block's amax; multiplying by s_enc before E4M3 casting guarantees the FP8 block scale itself does not overflow 448. The key advantage over MXFP4 (block 32, power-of-two E8M0 scale) is the smaller block (16) plus a higher-resolution E4M3 scale, which collectively minimizes block error rather than snapping to the nearest 2^n.

NVIDIA TensorRT Model Optimizer (modelopt) performs PTQ via mtq.quantize(model, config, forward_loop), where config = mtq.NVFP4_DEFAULT_CFG (also NVFP4_AWQ_LITE_CFG, NVFP4_KV_CFG, W4A8_NVFP4_FP8_CFG, and MSE/Hessian sweep variants). The NVFP4 numeric quantizer encodes the two-level scheme as num_bits=(2,1) with block_sizes={-1: 16} and scale_bits=(4,3) (the (4,3) = E4M3 block scale, plus the implicit per-tensor FP32 scale); these configs are loaded from YAML presets (configs/numerics/nvfp4, configs/ptq/presets/model/nvfp4). Modelopt is fake-quantization in PyTorch; calibration algorithms are passed via the config "algorithm" and include "max" (amax), MSE (NVFP4MSECalibrator), percentile, "smoothquant", and AWQ variants "awq_lite"/"awq_full"/"awq_clip", plus SVDQuant and GPTQ. Calibration uses a forward loop over ~128-512 representative samples (created via get_dataset_dataloader/create_forward_loop). Export uses modelopt.torch.export.export_hf_checkpoint(model, export_dir=...) producing a unified HuggingFace checkpoint (compressed-tensors-style) storing packed FP4 weights plus weight scales (per-group E4M3) and global FP32 scales, runnable in TensorRT-LLM/vLLM.

llm-compressor (vLLM project) applies NVFP4 with QuantizationModifier(targets="Linear", scheme="NVFP4", ignore=["lm_head"]) inside a oneshot(...) call over a calibration dataset, then model.save_pretrained(..., save_compressed=True). The scheme is defined in compressed-tensors quant_scheme.py. NVFP4 (W4A4) sets both weights and input_activations to num_bits=4, type=FLOAT, strategy=TENSOR_GROUP, symmetric=True, group_size=16, scale_dtype=FP8_E4M3; weights are dynamic=False (static, computed at calibration) while input_activations use dynamic=DynamicType.LOCAL with observer="static_minmax" — i.e. a STATIC per-tensor global activation scale (calibrated) but DYNAMIC per-group-of-16 local activation scales computed online at inference. NVFP4A16 is the weight-only variant (weights only, activations stay 16-bit). The global scale function generate_gparam computes global_scale = scale_data.max * quant_data.max / max_val_pos = 448*6/amax, identical to the paper.

For W4A4 specifically, activation outliers are the dominant accuracy problem because activations are quantized to 4 bits online. The two main remedies are (a) rotation/incoherence processing — QuaRot (fixed randomized Hadamard transforms) and SpinQuant (learned/optimized orthogonal rotations via Cayley) rotate weights and activations into a near-Gaussian space so outliers spread across channels — and (b) SmoothQuant, which migrates difficulty from activations to weights via per-channel scaling. A crucial format-specific finding ("Bridging the Gap..." paper) is that, unlike MXFP4, rotations can HURT NVFP4 when combined with plain round-to-nearest; the recommended practice is to match the Hadamard size to the NVFP4 group size (Had16 for NVFP4, vs larger Had128 for MXFP4's group-32). Rotation is applied BEFORE micro-block grouping (weight rotations fused/pre-applied offline, activation rotations done online), and the rotated tensor is then fed through the normal two-level (global FP32 + E4M3 block) scaling. For NVFP4, QuaRot and SpinQuant perform comparably (~74% vs ~73.7% recovery), whereas SpinQuant is clearly better for MXFP4.

Static vs dynamic: weights are always quantized statically (offline) — both their per-group E4M3 scales and per-tensor FP32 global scale are baked into the checkpoint. Activations in full NVFP4 use a hybrid: the per-tensor FP32 global activation scale is calibrated/static (this is what the small calibration set is for), but the per-16-element block (local) activation scales are computed dynamically at inference time from the live amax of each block — the Blackwell 5th-gen Tensor Core hardware natively performs this micro-scaling (grouping, dynamic per-block scaling, and the 4-bit matmul) during the GEMM. Weight-only schemes (NVFP4A16) skip activation quantization entirely and need no activation calibration.

**Gotchas:**
- The per-block scale is NOT stored as amax_block/6 directly; it is first multiplied by the per-tensor global encode scale s_enc and only THEN cast to E4M3 (s_dec,b,e4m3 = e4m3(s_dec,b * s_enc)). This multiplication is what guarantees the FP8 block scale stays within 448 and is why the two levels are coupled rather than independent. Quantizers that store amax_block/6 directly are wrong.
- Rotation (QuaRot/SpinQuant) is helpful for MXFP4 but can REDUCE NVFP4 accuracy when combined with plain round-to-nearest. Do not assume Hadamard rotation is always beneficial for FP4 — for NVFP4 match the rotation size to the group size (16) and consider format-aware rounding.
- Static vs dynamic is split per scale level, not per tensor: for activations the per-TENSOR FP32 global scale is STATIC (calibrated offline, the only reason a calibration set is needed), while the per-16-element LOCAL scales are DYNAMIC (computed online from each block's amax by Blackwell hardware). Weights are fully static.
- NVFP4 (W4A4) vs NVFP4A16 are different schemes: NVFP4A16 is weight-only (activations stay 16-bit, no activation calibration), while NVFP4 quantizes both. Confusing them changes whether activation outlier handling / calibration data is even needed.
- modelopt configs (NVFP4_DEFAULT_CFG etc.) are not Python literals — they are loaded from YAML/JSON presets at runtime, so reading config.py alone will not show num_bits/block_sizes/scale_bits values; you must read the preset files under configs/numerics and configs/ptq/presets.
- E4M3 max is 448 (the FP8 used for the block scale), not 240 or 416; and the FP4 max is 6. Both constants appear together as the product 6*448 in the global scale denominator — using only one of them is a common error.
- compressed-tensors stores the per-group scale in float8_e4m3 and the global scale in FP32; the dequant is value = code_E2M1 * block_scale_E4M3 * global_scale_FP32. Forgetting the global (second) scale silently truncates dynamic range.
- MXFP4 (block 32, E8M0 power-of-two scale) and NVFP4 (block 16, E4M3 scale + FP32 global) are frequently conflated; modelopt block_size for MX is 32 while NVFP4 is 16, and only NVFP4 has the FP32 per-tensor global scale.

### Low-bit (4-bit / W4A4) post-training quantization for Vision Transformers and vision encoders: why ViT activations break, key methods and accuracy, mixed-precision recipes, CLS/register-token outliers, and whether per-block (NVFP4 group=16) scaling helps

ViTs are much harder to quantize at 4 bits than CNNs, and the difficulty is overwhelmingly on the ACTIVATION side, not weights. Three pathological distributions recur across every paper: (a) post-LayerNorm activations have severe inter-channel variation, with some channel ranges exceeding ~40x the median, so a single per-tensor scale either clips the wide channels or wastes resolution on the narrow ones; (b) post-Softmax attention probabilities follow a power-law (almost all mass near 0, a tiny fraction near 1), so uniform quantizers spend codes on the dense low-value region and destroy the few large attention weights that actually route information; (c) post-GELU activations are strongly asymmetric (a dense band of small/negative values plus sparse large positives), which uniform symmetric quantizers handle badly. Empirically, the result is catastrophic: FQ-ViT-style uniform PTQ collapses to ~0.1% top-1 at W4A4, and naive MXFP4/NVFP4 PTQ shows >10% drops; even the early ViT-specialized methods (PTQ4ViT, APQ-ViT) only reach 30-68% at W4A4 versus ~81-85% FP. The "dense information" nature of vision tokens (every patch carries signal, unlike sparse/redundant text tokens) means there is little redundancy to absorb quantization noise, and per-image-varying background "sink"/high-norm tokens create activation outliers that move around, unlike LLMs where outliers sit on a fixed closed-set vocabulary (BOS/SEP).

The method lineage is well documented with concrete ImageNet numbers (all on ViT-S 81.39 / ViT-B 84.54 / DeiT-S 79.85 / DeiT-B 81.80 / Swin-S 83.23 / Swin-B 85.27 FP baselines). FQ-ViT (IJCAI'22) introduced Power-of-Two Factor for LayerNorm and Log-Int-Softmax but targeted W8A8 (with 4-bit attention maps), and fails at full W4A4. PTQ4ViT (ECCV'22) added twin uniform quantization for GELU/softmax plus a Hessian-guided scale metric, reaching near-lossless W8A8 but only ~30-76% at W4A4. APQ-ViT (ACM-MM'22) added blockwise calibration and Matthew-effect-preserving softmax quantization, ~+5% over PTQ4ViT at W4A4. NoisyQuant (CVPR'23) adds a fixed uniform noisy bias to reshape heavy-tailed activations (+1.7/1.1/0.5% at A6 for ViT/DeiT/Swin). RepQ-ViT (ICCV'23) is the key W4A4 breakthrough: it quantizes with per-channel (LayerNorm) and log-sqrt2 (softmax) quantizers, then mathematically reparameterizes to hardware-friendly per-layer/log2 for inference, lifting W4A4 to 65.05/68.48/57.43(DeiT-T)/69.03/75.61/79.45/78.32. RepQuant (2024) adds learnable per-channel dual clipping and pushes W4A4 to ViT-S 73.28, ViT-B 77.84, DeiT-S 75.21, DeiT-B 78.46, Swin-S 81.52, Swin-B 82.80 (+2.8 to +9.4 over RepQ-ViT). DopQ-ViT (2024) adds a Tan quantizer for softmax and a Median-based optimal scale factor (MOSF) for the LayerNorm reparameterization, reaching ViT-S 75.69 / ViT-B 80.95 / DeiT-S 75.84 / DeiT-B 80.13 / Swin-S 81.71 / Swin-B 83.34 at W4A4 (W6A6 is essentially lossless, within ~1%). I-ViT (ICCV'23) is integer-only (Shiftmax/ShiftGELU/dyadic), matching FP at INT8 with 3.7-4.1x GPU speedup but not a 4-bit method.

Mixed-precision practice is consistent across the literature. Softmax and LayerNorm are usually left in higher precision or handled with special non-uniform quantizers rather than naive INT4 (PTQ4ViT explicitly leaves softmax/norm unquantized). Importance/sensitivity studies (Mix-QViT, LRP-QViT, Q-ViT) find the projection layer, the MSA/GELU blocks, and early blocks most sensitive, so they get higher bits, while deeper blocks can be pushed lower. The attention-score matmul (Q*K and the post-softmax A*V) is the most fragile matmul. NVIDIA's 2026 NVFP4 production recipe for hybrid/transformer models keeps attention layers plus the first and last two blocks in BF16 and quantizes the rest to NVFP4 - the same first/last-block + attention-in-higher-precision pattern seen in ViT PTQ. The classification head and patch-embedding conv are commonly kept higher precision because they are single sensitive layers with little compute cost.

CLS/register-token outliers are a first-class problem. DINOv2 develops high-norm artifact tokens (~10x norm, ~2% of tokens) on uninformative background patches; "Vision Transformers Need Registers" (ICLR'24) shows adding register tokens absorbs these outliers, and notably the register norms appear "quantized" (much tamer). For quantization this matters directly: a handful of extreme-norm tokens blows up any per-tensor activation scale, so per-token / per-block scaling is far more robust than per-tensor. The 2025 paper "Activation Quantization of Vision Encoders Needs Prefixing Registers" (RegCache, arXiv 2510.04547) shows vision-encoder outliers arise in MIDDLE layers (blocks ~5-8) on per-image-varying background tokens - unlike LLMs' fixed-vocab sink tokens - and that suppressing them (reducing CLIP max token norm 41.38->11.45) yields large low-bit gains: up to +44.5% at W4A4 and +11.2% average at W6A6 on zero-shot ImageNet (though absolute W4A4 zero-shot CLIP remains very low, ~1.8%, showing vision-language encoders at W4A4 are still essentially broken without QAT).

On NVFP4 specifically: per-block (group=16) scaling DOES help the ViT outlier problem relative to per-tensor, because a localized outlier only consumes the dynamic range of its own 16-element block rather than the whole tensor, and the two-level scheme (per-block E4M3 + per-tensor FP32) gives non-power-of-two scales and wide range. BUT there are two important caveats from primary sources. First, NVIDIA's own 2026 QAD report states that "common PTQ algorithms often fail to improve over baseline NVFP4 performance because the small block size neutralizes traditional outlier mitigation techniques" - i.e., smoothing/rotation tricks designed for per-tensor lose their leverage when scaling is already fine-grained, and PTQ trails QAD by 20-30% with QAD recovering >95%. Second, "Four Over Six" (Dec 2025) shows that within a block the dominant FP4 error is NOT outliers (those just consume scale range with little final-loss impact) but the non-uniform mid-range step sizes of the FP4 grid: values at 66-100% of the block max have no good representation, which adaptive M=4/M=6 block scaling fixes. Crucially, none of the NVFP4-specific outlier/sensitivity papers (Four Over Six, the layer-wise/block-wise diagnosis, QAD) actually evaluate Vision Transformers - they are LLM/VLM-centric - so ViT-native NVFP4 PTQ remains an open, under-evaluated area, with the closest ViT evidence being MXFP4 (block-32) TRAINING work (TetraJet, Feb 2025) confirming naive MXFP4 PTQ on ViTs is impractical and that QAT/oscillation control is needed to reach near-FP accuracy.

**Gotchas:**
- No published NVFP4-specific PTQ study actually evaluates Vision Transformers - Four Over Six, the layer/block sensitivity diagnosis, and the NVIDIA QAD report are all LLM/VLM-centric. The closest ViT-native FP4 evidence is MXFP4 (block-32) TRAINING (TetraJet), not NVFP4 PTQ. ViT-native NVFP4 PTQ is genuinely under-evaluated as of mid-2026.
- Per-block scaling is NOT a free fix for the ViT outlier problem. NVIDIA explicitly reports that the small 16-element block 'neutralizes traditional outlier mitigation techniques' (SmoothQuant/rotation-style methods lose leverage), so existing per-tensor-era PTQ tricks may not transfer to NVFP4.
- The dominant FP4 error per 'Four Over Six' is mid-range non-uniform step sizes, NOT outliers - so framing the ViT problem purely as 'outliers' is incomplete for FP4; the power-law softmax problem (values near the block max needing fine resolution) maps directly onto FP4's coarse high-range steps.
- Different papers use slightly different FP baselines (e.g., DeiT-B 81.80 vs 81.85, Swin-B 85.27 vs 83.60 in FQ-ViT's table, and TetraJet uses its own 90-epoch-trained baselines like DeiT-S 73.33 not the timm 79.85). Do not cross-compare absolute numbers across papers without checking the baseline.
- W4A4 'success' is relative: RepQuant/DopQ-ViT reach high-70s/low-80s on classification ViTs, but vision-LANGUAGE encoders (CLIP zero-shot) at W4A4 are still essentially broken (~1.8% even with RegCache), so 'W4A4 works for ViTs' holds only for supervised ImageNet classifiers, not vision encoders broadly.
- Most 'W4A4' ViT PTQ methods do NOT actually run softmax and LayerNorm at INT4 - they either leave them unquantized (PTQ4ViT) or use special non-uniform/log/power-of-two quantizers; the W4A4 label refers to the linear/matmul weights and activations. A true fully-INT4 ViT (including softmax/norm) is a stronger and rarer claim.
- FQ-ViT's '4-bit' refers to 4-bit attention MAPS within an otherwise W8A8 model, not W4A4 weights+activations. It is frequently miscited as a W4A4 method (where it scores ~0.1%).
- I-ViT, MX/NVFP4 training works, and QAD are not PTQ - mixing them into a 'W4A4 PTQ' comparison conflates training-time and post-training methods. I-ViT is INT8-only.

### Quantization-relevant architectural details of SigLIP/SigLIP2, Qwen3-VL vision encoder, DINOv2/DINOv3, and V-JEPA2/2.1 vision encoders (norms, activations, attention layout, tokens, patch embed, positional embeddings, dims, GEMM tensors, numeric-stability quirks)

All five families are standard pre-norm ViTs (LayerNorm before attn and MLP, residual after), with one exception: DINOv3 ViT and V-JEPA2 still use LayerNorm, not RMSNorm. None of these vision towers use RMSNorm in the encoder blocks — RMSNorm appears only in the *text/LLM* side of Qwen3-VL (rms_norm_eps=1e-6). LayerNorm eps is 1e-6 for SigLIP/SigLIP2/Qwen3-VL-vision/DINOv2/V-JEPA2 and 1e-5 for DINOv3 ViT. This matters for NVFP4/FP4 work because the LayerNorm is the natural place to absorb/insert per-tensor scale and the eps differences affect where you can fold scales.\n\nThe big GEMMs in every block are the same four logical tensors: QKV projection, attention output projection, MLP fc1 (up), MLP fc2 (down). QKV fusion differs by family and directly affects how you tile/quantize: Qwen3-VL vision FUSES qkv into one `nn.Linear(dim, 3*dim, bias=True)` and uses `proj` for output; SigLIP/SigLIP2 use SEPARATE `q_proj/k_proj/v_proj` + `out_proj`; DINOv2 uses SEPARATE `query/key/value` (Dinov2SelfAttention) + `dense` output (Dinov2SelfOutput); DINOv3 ViT uses SEPARATE `q_proj/k_proj/v_proj` with independent bias flags (query_bias=True, key_bias=False, value_bias=True) + `o_proj`; V-JEPA2 uses SEPARATE `query/key/value` + `proj`. All are vanilla multi-head attention (no GQA) in the vision towers; GQA/MQA only appears optionally in the Qwen3-VL *text* decoder. Activations: SigLIP, SigLIP2, and Qwen3-VL vision all use gelu_pytorch_tanh (tanh-approx GELU); DINOv2 default and V-JEPA2 use plain gelu; DINOv2-large checkpoint uses use_swiglu_ffn=false (the 'large' on HF is the non-SwiGLU MLP variant). DINOv3 has a configurable gated SwiGLU MLP (`use_gated_mlp`, gate_proj/up_proj/down_proj) vs plain MLP (up_proj/down_proj) — large DINOv3 variants (ViT-7B/16) use SwiGLU. MLP layer names: SigLIP/V-JEPA2 use fc1/fc2; Qwen3-VL vision uses linear_fc1/linear_fc2; DINOv2 uses fc1/fc2 (or weights_in/weights_out for SwiGLU); DINOv3 uses up_proj/down_proj (+ gate_proj when gated).\n\nToken/positional structure varies and is a quantization gotcha for outlier handling: SigLIP/SigLIP2 have NO CLS token and NO register tokens — they pool with a learned-probe multihead attention pooling head (SiglipMultiheadAttentionPoolingHead, `self.probe`), and use learned interpolatable positional embeddings. DINOv2 has 1 CLS token + learned interpolatable position_embeddings; DINOv2-with-registers adds 4 register tokens. DINOv3 has 1 CLS token + 4 register tokens (num_register_tokens=4) and uses RoPE (DINOv3ViTRopePositionEmbedding, rope_theta=100.0) applied only to patch tokens (no learned absolute pos embed). Qwen3-VL vision has NO CLS/register tokens; it uses a hybrid of learned absolute position embeddings (nn.Embedding, bilinearly interpolated) PLUS 2D rotary (Qwen3VLVisionRotaryEmbedding). V-JEPA2 has NO CLS token (uses an attentive pooler with learned query tokens) and 3D-RoPE over (depth/time, height, width). CLS and register tokens are high-norm 'attention sink' tokens — they carry large-magnitude activations and are classic outlier sources you must keep in higher precision or scale separately.\n\nPatch/tubelet embedding: SigLIP/SigLIP2/DINOv2/DINOv3 use a 2D Conv (Conv2d, patch_size 14 for SO400M/DINOv2, 16 for SigLIP2-base/DINOv3). Qwen3-VL vision and V-JEPA2 use 3D Conv (Conv3d) over tubelets: Qwen3-VL kernel/stride = [temporal_patch_size=2, patch=16, patch=16]; V-JEPA2 kernel = (tubelet_size=2, 16, 16). LayerScale (a learned per-channel multiply, lambda1) is present in DINOv2 (layerscale_value=1.0) and DINOv3 (DINOv3ViTLayerScale) after both attn and MLP — this is a per-channel affine that can be folded but creates per-channel dynamic range. SigLIP/SigLIP2/Qwen3-VL-vision/V-JEPA2 have NO layerscale. Attention logit scaling is the standard head_dim**-0.5 everywhere (SigLIP `self.scale = head_dim**-0.5`); none of these use the Qwen2-VL-style learned attention temperature in the vision tower. SigLIP's contrastive logit_scale/logit_bias (bias init -10) live in the CLIP wrapper, not the vision encoder GEMMs.\n\nConcrete dims (encoder hidden/intermediate/layers/heads): SO400M-class (original SigLIP-SO400M, SigLIP2-SO400M, Qwen3-VL-MoE vision) all share the shape-optimized SoViT-400M backbone: hidden=1152, intermediate(MLP)=4304 (mlp_ratio≈3.74, an unusual non-4x ratio from the 'Getting ViT in Shape' scaling-law design), layers=27, heads=16, head_dim=72. SigLIP2-base: hidden=768, intermediate=3072, layers=12, heads=12, patch=16. Qwen3-VL vision: depth=27, hidden=1152, intermediate=4304, heads=16, patch=16, out_hidden_size=2048 (30B-A3B) projected via a 2-layer merger, spatial_merge_size=2 (4 patches merged → token count /4), deepstack injects ViT layers [8,16,24] into the LLM. DINOv2-large: hidden=1024, layers=24, heads=16, patch=14, image=518. DINOv2-base: hidden=768, layers=12, heads=12, head_dim=64. DINOv3-ViT-S (config default): hidden=384, layers=12, heads=6, patch=16, image=224; ViT-B/16 hidden=768. V-JEPA2 ViT-g (vjepa2-vitg-fpc64-256): hidden=1408, layers=40, heads=22, mlp_ratio≈4.36 (intermediate≈6144), patch=16, tubelet=2, crop=256, 64 frames; predictor is a small ViT (hidden=384, 12 layers, 12 heads). V-JEPA 2.1 (arXiv 2603.14482, 2026) scales the encoder to ViT-G (~2B), adds modality-specific 2D/3D tokenizers feeding a shared encoder, a dense predictive loss, and a 384x384 high-res cooldown.

**Gotchas:**
- No vision tower here uses RMSNorm — all use LayerNorm in the encoder blocks. RMSNorm only appears in the Qwen3-VL TEXT/LLM decoder. Do not assume RMSNorm from the 'Qwen3' name.
- LayerNorm eps differs: 1e-6 for SigLIP/SigLIP2/Qwen3-VL-vision/DINOv2/V-JEPA2, but 1e-5 for DINOv3 ViT. This affects scale-folding into the norm.
- QKV fusion is NOT uniform. Qwen3-VL vision fuses qkv (one Linear, 3*dim out). SigLIP/SigLIP2/DINOv3 keep separate q/k/v. DINOv2/V-JEPA2 keep separate query/key/value. Output-proj names differ: out_proj (SigLIP), proj (Qwen3-VL, V-JEPA2), dense (DINOv2), o_proj (DINOv3). Your weight-mapping/quant config must branch per family.
- DINOv3 splits the attention bias asymmetrically: query_bias=True, key_bias=False, value_bias=True. A naive 'all-or-nothing qkv bias' assumption will break DINOv3.
- SigLIP and Qwen3-VL vision use gelu_pytorch_tanh (tanh-approx GELU), while DINOv2/V-JEPA2 use exact gelu. The tanh approx changes the activation distribution feeding fc2 — relevant if you calibrate FP4 scales per-activation.
- CLS and register tokens are high-norm outlier/attention-sink tokens (DINOv2: 1 CLS; DINOv3: 1 CLS + 4 registers; SigLIP/SigLIP2/Qwen3-VL-vision: none, use attention-pool probe; V-JEPA2: none, attentive pooler). These special tokens carry large activations and are a known FP4/INT4 outlier hazard.
- LayerScale (per-channel learned multiply, lambda1) is present in DINOv2 and DINOv3 after BOTH attn and MLP residual branches. It creates strong per-channel dynamic range and can be folded into the preceding GEMM output scale, but must not be ignored. SigLIP/SigLIP2/Qwen3-VL-vision/V-JEPA2 have NO layerscale.
- DINOv2 'large' on HF (facebook/dinov2-large) has use_swiglu_ffn=false (plain fc1/fc2 MLP). Do not assume SwiGLU for DINOv2; the original facebookresearch giant (ViT-g) variant is the one that used swiglufused. DINOv3, by contrast, uses gated SwiGLU (gate_proj/up_proj/down_proj) for its large variants via use_gated_mlp.
- Qwen3-VL vision merger and MLP both reuse the name 'linear_fc1/linear_fc2' but the MLP is plain (fc1->gelu_tanh->fc2) and the merger is fc1->GELU->fc2 projecting to out_hidden_size; neither is a gated GEGLU despite some doc summaries calling the merger 'gated'. Verified directly from source.
- SO400M backbone (original SigLIP-SO400M, SigLIP2-SO400M, Qwen3-VL vision) has MLP intermediate=4304 with hidden=1152, i.e. mlp_ratio≈3.74 (NOT 4x). Hardcoding 4*hidden for the MLP GEMM shape will be wrong for these. head_dim=1152/16=72 (not 64).
- Qwen3-VL vision spatial_merge_size=2 means 2x2=4 neighboring patches are merged before the LLM, cutting visual token count by 4x; the merger GEMM input dim is hidden_size*merge^2. Sequence length is dynamic (NaFlex/grid_thw), not fixed.
- Patch embedding is Conv3d for Qwen3-VL vision and V-JEPA2 (tubelet over time), but Conv2d for SigLIP/SigLIP2/DINOv2/DINOv3. The Conv weights are small relative to the linear GEMMs and usually left in higher precision.
- Attention logit scale is the vanilla head_dim**-0.5 in all five vision towers (SigLIP: self.scale=head_dim**-0.5). The SigLIP contrastive logit_scale/logit_bias (bias init -10) belong to the CLIP head, not the encoder GEMMs — don't confuse them.
- V-JEPA2 config DEFAULT (hidden_size=1024, 24 layers) is a ViT-L placeholder; the released ViT-g checkpoint overrides to hidden=1408/40 layers/22 heads. Always read the actual checkpoint config.json, not the class defaults.
- V-JEPA 2.1 numbers come from a 2026 arXiv preprint (2603.14482); exact released HF config tensor names for the ViT-G 2B encoder were not confirmed from a primary config.json in this pass — treat 2.1-specific dims as paper-stated until the HF checkpoint config is checked.

### NVFP4 W4A4 quantized-linear integration for PyTorch HuggingFace/timm transformers on sm_120

NVFP4 W4A4 means E2M1 4-bit weights AND E2M1 4-bit activations, both block-scaled (16-element micro-blocks with a per-block E4M3 FP8 scale) plus a per-tensor FP32 global scale. Three production toolkits implement the nn.Linear swap, and they converge on the SAME math and the SAME hardware GEMM. (1) torchao is the cleanest, most reusable reference for a from-scratch build: `quantize_(model, NVFP4DynamicActivationNVFP4WeightConfig(...))` runs `_nvfp4_inference_linear_transform` (registered via `@register_quantize_module_handler`) which replaces each `nn.Linear.weight` with an `NVFP4Tensor` (a tensor subclass) via `NVFP4Tensor.to_nvfp4(...)`. The subclass overrides `aten.linear.default`/`F.linear` -> `nvfp4_linear`, and at runtime quantizes the activation (dynamic per-tensor amax -> global scale, then per-block E4M3) and calls `torch._scaled_mm(a.qdata.view(torch.float4_e2m1fn_x2), b.qdata.view(torch.float4_e2m1fn_x2), a_scale_blocked.view(torch.float8_e4m3fn), b_scale_blocked.view(torch.float8_e4m3fn), ...)`. modelopt and compressed-tensors instead insert observer/quantizer modules: modelopt patches `nn.Linear.forward` with `TensorQuantizer` (input + weight quantizers); compressed-tensors swaps to `CompressedLinear`. All three serialize the same 4 tensors per weight (packed fp4, per-block E4M3 scale, FP32 global scale, plus an activation global scale).

(2) The Python-callable FP4 GEMMs are: `torch._scaled_mm` (PyTorch native, wired to the cuBLAS/cuBLASLt nvfp4 kernel when operands are `float4_e2m1fn_x2` and scales are `float8_e4m3fn`, PR #148792); torchao's own ops (`nvfp4_linear` dispatching to `_scaled_mm`); `flashinfer.gemm.mm_fp4(a, b, a_descale, b_descale, alpha=None, out_dtype=torch.bfloat16, block_size=16, use_8x4_sf_layout=False, backend='auto'|'cutlass'|'cudnn'|'trtllm'|'cute-dsl'|'b12x')`; vLLM `_custom_ops`/`cutlass_scaled_fp4_mm` and its `--linear-backend` selector; and raw CUTLASS examples (`72b_blackwell_nvfp4_nvfp4_gemm.cu`). All quantize activations with a helper like FlashInfer's `nvfp4_quantize(a, a_global_sf, sfLayout=SfLayout.layout_128x4)` or torchao's `NVFP4Tensor.to_nvfp4`.

(3) Calibration in practice (modelopt is the canonical loop): `mtq.quantize(model, config, forward_loop)` inserts `TensorQuantizer` modules into Linear/Conv layers and patches their forward to fake-quant, then runs the user `forward_loop(model)` (iterate calibration batches, `model(batch)` under `torch.no_grad`). With `"calibrator": "max"`, forward hooks collect per-tensor `amax` via `reduce_amax()` during the loop; afterward `max_calibrate` folds amax into the `_scale`/global-scale buffers. For NVFP4 only the per-tensor activation global scale needs calibration data; the per-block (size-16) activation scales are computed dynamically at inference. compressed-tensors mirrors this with `apply_quantization_config()` + calibration + `freeze_module_quantization()`.

(4) There is NO ready-made ViT-specific or general-purpose (non-LLM) NVFP4 PTQ toolkit you can drop in. The well-known ViT PTQ repos (PTQ4ViT, FQ-ViT) are INT8/uniform, not NVFP4, and unmaintained for Blackwell. modelopt is the closest general toolkit and DOES quantize ViT Linear layers (there is a working `torch_quant_to_onnx.py` ViT example and FLUX/SDXL/LTX-2 diffusion fp4 examples), but its NVFP4 ONNX PTQ path is reported to fail on pure CNN vision models (ResNet50, ConvNeXt; issue #362). torchao's NVFP4 inference workflow is model-agnostic (`quantize_` works on any module tree of `nn.Linear`), so the realistic plan is to reuse torchao's `NVFP4Tensor` + `_scaled_mm` path (or modelopt's quantizer/calibration) and apply it to a timm/HF ViT yourself; you will essentially be building the ViT NVFP4 wrapper from scratch on top of these LLM-oriented primitives.

(5) Packing/layout: weights are E2M1 packed two-per-byte: `(q[...,1::2] << 4) | q[...,0::2]` (modelopt) / torchao `pack_uint4`, exposed as `torch.float4_e2m1fn_x2`. Per-block scales are `torch.float8_e4m3fn` (UE4M3, max 448), one per 16 elements. The GEMM does NOT accept row-major scales: it requires the swizzled 128x4 scale-factor layout (torchao `to_blocked()` rearranges 128x4 blocks into 32x16 tiles; FlashInfer `SfLayout.layout_128x4`). Global scale is FP32. The two implementations agree on the formula: global_scale = amax / (F4_E2M1_MAX * F8E4M3_MAX) = amax / (6.0 * 448.0); per-block scale = block_amax / 6.0 clamped to [2**-9, 448]; dequant = q_e2m1 * block_scale_e4m3 * global_scale_fp32.

- (high) NVFP4 = E2M1 (1 sign, 2 exp, 1 mantissa), representable {0,0.5,1,1.5,2,3,4,6}, max 6.0; micro-block 16 values; per-block scale E4M3 FP8 (max 448); per-tensor scale FP32. Dequant x = x_q * block_scale * global_scale. Produced by TensorRT Model Optimizer and LLM Compressor. — https://developer.nvidia.com/blog/introducing-nvfp4-for-efficient-and-accurate-low-precision-inference/
- (high) No NVFP4 ViT/general-purpose PTQ toolkit exists; known ViT PTQ repos PTQ4ViT and FQ-ViT are INT8 uniform quantization (near-lossless at 8-bit), not FP4, and predate Blackwell. — https://github.com/hahnyuan/PTQ4ViT
- (high) modelopt NVFP4 quantizes ViT/transformer Linear layers (working torch_quant_to_onnx.py ViT example, plus quantize.py --format fp4 diffusion examples for FLUX/SDXL/LTX-2), but NVFP4 ONNX PTQ is reported to consistently fail on pure CNN vision models (ResNet50, MobileNetv5, ConvNeXt). — https://github.com/NVIDIA/Model-Optimizer/issues/362
- (high) All FlashInfer NVFP4 mm_fp4 backends were reported broken on SM120 (FlashInfer 0.6.3): cutlass returns all zeros, cudnn raises cudnnGraphNotSupportedError, trtllm raises BackendSupportedError 'does not support backend trtllm with capability 120'. Quantization step works; only GEMM backends fail. — https://github.com/flashinfer-ai/flashinfer/issues/2577
- (high) PyTorch auto-detected CUDA arch flags strip the 'a'/'f' suffix (sm_120a -> sm_120), breaking Blackwell NVFP4/block-scaled MMA which need the architecture-specific sm_120a; workaround is TORCH_CUDA_ARCH_LIST=12.0a (issue #172807, fixed by PR #172721). — https://github.com/pytorch/pytorch/issues/172807
- (medium) vLLM 0.13.0 ships SM120-specific NVFP4 kernels (scaled_mm_sm120_kernels, blockwise_moe); requires vLLM 0.12.0+ and VLLM_USE_FLASHINFER_MOE_FP4=1; backends selectable via --linear-backend (flashinfer-b12x/cudnn/trtllm/cutlass, cutlass, marlin, emulation). On SM120 'auto' prefers b12x (NVFP4 only) then cutlass then cudnn. — https://discuss.vllm.ai/t/sm120-rtx-pro-6000-nvfp4-moe-performance-report-qwen3-5-397b/2536

**Gotchas:**
- sm_120 (consumer/prosumer Blackwell: RTX 5090, RTX PRO 6000) is architecturally DIFFERENT from sm_100 (B200/GB200): no tcgen05.mma, no Tensor Memory. SM100 FP4 GEMM kernels will NOT run on SM120. You must use SM120-specific kernels (warp-level mma.sync m16n8k64).
- NVFP4/block-scaled MMA requires the architecture-specific 'sm_120a' target, not plain 'sm_120'. PyTorch's auto arch detection silently strips the 'a' suffix and breaks NVFP4. Build/run with TORCH_CUDA_ARCH_LIST=12.0a (pytorch issue #172807).
- FP4 GEMM backends have been broken/flaky on SM120: FlashInfer cutlass backend returned all zeros, cudnn raised cudnnGraphNotSupportedError, trtllm rejected capability 120 (FlashInfer 0.6.3). Verify the GEMM on your exact GPU before trusting it; do not assume torch._scaled_mm fp4 'just works' on sm_120.
- The hardware GEMM does NOT accept row-major scale tensors. Scales must be pre-swizzled to the 128x4 layout (torchao to_blocked / FlashInfer SfLayout.layout_128x4). Passing un-swizzled scales gives wrong results, not an error.
- Block size is 16 for NVFP4 (vs 32 for MXFP4), so the contraction (K) dimension of every quantized Linear must be divisible by 16; scale rows are padded to 128. ViT MLP/attention projection dims are typically divisible by 16, but verify per-layer.
- NVFP4 is for nn.Linear (block-scaled along K). It does not apply to Conv2d patch-embed / CNN stems well: modelopt NVFP4 ONNX PTQ fails on CNN vision models, and conv activation ranges are too narrow for FP4's coarse granularity. Keep the patch-embed conv, LayerNorm, and the classification head in higher precision.
- No turnkey ViT or general-purpose NVFP4 PTQ toolkit exists. Existing ViT PTQ repos (PTQ4ViT, FQ-ViT) are INT8, not FP4. You will reuse torchao's NVFP4Tensor/_scaled_mm primitives or modelopt's quantizer+calibration and write the ViT wrapper yourself.
- W4A4 needs activation calibration: only the per-tensor activation GLOBAL scale is calibrated from data (forward hooks collecting amax); the per-block size-16 activation scales are computed dynamically at inference. Forgetting the global activation scale calibration degrades accuracy.
- Checkpoint format mismatch across toolkits: compressed-tensors stores {weight_packed, weight_scale, weight_global_scale, input_global_scale} while modelopt/vLLM expect {weight, weight_scale_inverse, input_scale}. Pick one ecosystem; cross-loading needs a converter.
- torchao NVFP4 lives under torchao.prototype.mx_formats (prototype/unstable API); class and function names (NVFP4DynamicActivationNVFP4WeightConfig, to_nvfp4, to_blocked) can change between versions, so pin the torchao version.
